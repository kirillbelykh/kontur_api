# Kontur API как локальный агент для WMS

## Задача

Сохранить `kontur_api` как локальную Windows-утилиту, но использовать ее не только как отдельное приложение, а и как исполняющий агент для нового WMS-раздела `Честный знак`.

## Ключевой принцип

`kontur_api` остается на каждом ПК и отвечает за все локальные зависимости:

- cookies и браузерную авторизацию
- CryptoPro / CAdES
- сетевые вызовы в Честный Знак и Контур
- печать через BarTender
- локальные файлы, очереди, историю и runtime

WMS при этом берет на себя только пользовательскую оболочку и бизнес-маршрутизацию.

## Режимы работы

### 1. Текущий отдельный режим

Пользователь запускает `ui_v2/main_v2.py` и работает в отдельном окне PyWebView.

### 2. Новый встроенный режим

При запуске `ui_v2/main_v2.py` дополнительно поднимается локальный embed-сервер:

- `http://127.0.0.1:8787`
- опционально `https://127.0.0.1:8788`

Этот сервер:

- отдает интерфейс `ui_v2` в embed-режиме
- принимает запросы от WMS-страницы в браузере пользователя
- использует тот же `ApiBridge`, что и desktop-окно

## Переменные окружения

Минимальный набор для WMS-режима:

```env
WMS_BOUND_USER=ivanov
WMS_ALLOWED_ORIGINS=https://wms.grund-lage.ru,http://localhost:5173,http://127.0.0.1:5173
WMS_EMBED_SERVER_ENABLED=1
WMS_EMBED_SERVER_HOST=127.0.0.1
WMS_EMBED_SERVER_PORT=8787
WMS_EMBED_SERVER_HTTPS_ENABLED=0
WMS_EMBED_SERVER_HTTPS_PORT=8788
```

### Что означает каждая переменная

- `WMS_BOUND_USER`
  - логин пользователя WMS, для которого предназначен локальный агент на этом ПК
- `WMS_ALLOWED_ORIGINS`
  - список доменов, которым разрешено обращаться к локальному агенту из браузера
- `WMS_EMBED_SERVER_ENABLED`
  - включает локальный embed-сервер
- `WMS_EMBED_SERVER_HOST`
  - адрес прослушивания, по умолчанию только loopback
- `WMS_EMBED_SERVER_PORT`
  - HTTP-порт локального агента
- `WMS_EMBED_SERVER_HTTPS_ENABLED`
  - включает HTTPS-слой для локального агента
- `WMS_EMBED_SERVER_HTTPS_PORT`
  - HTTPS-порт локального агента

## Почему привязка идет через пользователя

В WMS одновременно могут работать Ваня, Аня и Таня.

Важно, чтобы:

- заказ кодов, который оформил Ваня, шел через утилиту Вани
- печать, которую нажала Аня, шла на ПК Ани
- неиспользуемый сегодня ПК Тани не получал команды по ошибке

Именно для этого агент на ПК сообщает WMS, к какому `username` он привязан.

## Что уже делает локальный embed-сервер

1. Поднимается вместе с `ui_v2/main_v2.py`.
2. Отдает `ui_v2` в embed-режиме по query-параметру `embed=1`.
3. Поддерживает начальный маршрут через `route=orders|download|...`.
4. Отдает `/api/health`, чтобы WMS могла проверить:
   - доступность агента
   - привязанного пользователя
   - готовность embed-режима
5. Ограничивает CORS разрешенными origin'ами.

## Что остается неизменным

1. `ApiBridge` продолжает быть единым исполняющим слоем.
2. Старое окно PyWebView продолжает работать.
3. Текущий `CHZ bridge` для WMS-заявок не ломается.
4. Отдельный запуск `ui_mobile/server_mobile.py` тоже остается возможным.

## План развития

### Ближайший этап

- WMS показывает новый раздел `Честный знак`
- пользователь открывает нужную вкладку внутри WMS
- контент приходит из локального агента в embed-режиме

### Следующий этап

- по одному переносим экраны из `ui_v2/ui/app.js` в React/WMS
- `kontur_api` остается локальным исполнителем без обязательного отдельного UI

### Дальняя цель

Оставить `kontur_api` как сервисный агент:

- авторизация
- CryptoPro
- Контур / ЧЗ API
- печать
- локальные задания

а пользовательский интерфейс полностью вести из WMS.

## Быстрый откат

Если встроенный режим временно отключаем:

1. WMS скрывает раздел `Честный знак`
2. пользователи продолжают запускать `kontur_api` отдельно
3. вся текущая логика продолжает работать без архитектурного разрыва
